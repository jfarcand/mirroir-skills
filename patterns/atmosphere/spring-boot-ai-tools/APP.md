# spring-boot-ai-tools

AI streaming chat **with `@AiTool` tool calling**. Same `@AiEndpoint` /
Atmosphere Console plumbing as `spring-boot-ai-chat`, plus four tools
(`get_current_time`, `get_city_time`, `get_weather`, `convert_temperature`)
declared with the framework-agnostic `@AiTool` annotation. Tool invocations
emit `ToolStart` / `ToolResult` `AiEvent` frames that render in the Console's
"AGENT COLLABORATION" panel (`[data-testid=tool-activity]`) тАФ this is the
must-pass surface that distinguishes this sample.

## Boot prerequisites

- JVM: JDK 21
- Default port: **8090** (set in `application.yml` тЖТ `server.port`; not 8080).
- Command: `./mvnw -q spring-boot:run -pl samples/spring-boot-ai-tools` from
  the Atmosphere repo root.
- Boot wait: ~30тАУ45 s; ready when TCP `:8090` accepts connections AND
  `GET /atmosphere/console/` returns 200 AND `GET /api/console/info` reports
  `"runtime":"demo"`.
- Env the SKILL.md must set:
  - `LLM_MODE=fake` тАФ routes through `FakeLlmClient` for deterministic
    simulated streaming + tool invocations.
  - `ATMOSPHERE_AUTH_ENABLED=false` тАФ same reason as the other AI samples.

## Console-info advertisement

`GET /api/console/info` returns:

```json
{
  "subtitle": "AI chat with tool calling visualization",
  "endpoint": "/atmosphere/ai-chat",
  "runtime": "demo",
  "mode": "ai"
}
```

`runtime: demo` + `mode: ai` matches `spring-boot-ai-chat`. The subtitle is
the runtime-truth signal that tools are wired (matches
`atmosphere.console-subtitle` in `application.yml`).

## Structure

### Header (banner)
- Heading: "Atmosphere AI Console"
- Subtitle: `"AI chat with tool calling visualization"`
- Build-version chip: `"v4"`

### Navigation tabs

Six tabs: Chat / **Sessions** / Policies (0) / Decisions / Commitments / OWASP.
Sessions is present (Spring Boot Console build, vs Quarkus's five-tab
omission).

### Chat tab (default)

Same selector vocabulary as `spring-boot-ai-chat`:
- `data-testid="status-label"` тАФ observed `"Connected ┬╖ websocket"` (transport
  segment present here, unlike the bare `"Connected"` seen earlier on Quarkus).
- `data-testid="chat-input"`, `data-testid="chat-send"`, `data-testid="message-bubble"`
  with `.message--user` / `.message--assistant` modifier classes.

What's different from a plain AI chat: the assistant turn includes a
tool-activity panel rendered **before** the assistant body. Observed via
chrome-devtools-mcp snapshot 2026-05-15 for prompt `"What time is it in Tokyo?"`:

- User bubble: `[data-testid=message-bubble].message--user` тАФ "U" / prompt / timestamp
- Tool-activity panel: `[data-testid=tool-activity]` тАФ header "AGENT COLLABORATION",
  one tool entry per call:
  - check mark "тЬУ" + tool display name "Get City Time" + status "Done"
  - args block: `"city: tokyo"`
  - result block: `{"status":"success"}`
- Assistant bubble: `[data-testid=message-bubble].message--assistant` тАФ
  `"[Tool: get_city_time] The time in tokyo is 08:32:16. This tool was
  registered with @AiTool and bridged to the active backend."`
  followed by token metrics: `"20 tokens ┬╖ 1191ms ┬╖ 16.8 tok/s"`.

Tool body is generated by `FakeLlmClient`'s tool-call simulator under
`LLM_MODE=fake`; real-LLM mode produces a different surface body but the
tool-activity panel shape (header / tool entries / args / result) is the same.

### Prompt тЖТ tool routing matrix (must-pass cohort)

- `"What time is it in Tokyo?"` тЖТ `get_city_time` (cheapest, used for SKILL.md)
- `"What time is it?"` тЖТ `get_current_time`
- `"What's the weather in Paris?"` тЖТ `get_weather`
- `"Convert 25C to Fahrenheit"` тЖТ `convert_temperature`

Any of these works; the SKILL.md picks one and sticks with it for
reproducibility.

## Obstacles

- `ATMOSPHERE_AUTH_ENABLED=true` тЖТ 401 on the console.
- Without `LLM_MODE=fake`, the sample probes for a real `LLM_API_KEY` and may
  surface a real-LLM tool-call body тАФ assistant copy varies, but the
  tool-activity panel still renders. The SKILL.md asserts structure
  (panel + bubble visible), not specific text.
- Port collision: the sample defaults to 8090 specifically to coexist with
  other AI samples on 8080. Don't override `server.port` to 8080 тАФ it would
  collide with `spring-boot-ai-chat` if that's running.

## Skip

- "Clear" button тАФ wipes session state mid-flow.
- Sessions / Policies / Decisions / Commitments / OWASP tabs тАФ empty.
- Multi-tool prompts ("Compare time in Tokyo and Paris") тАФ each invocation
  emits multiple tool entries which is correct but harder to assert
  deterministically on selector counts; use a single-tool prompt instead.

## Tips

- Use one of the prompt-routing exemplars above; the SKILL.md text is
  load-bearing for the routing assertion if you want to verify a specific
  tool. If you only assert "any tool fired", any of the four prompts works.
- Tool-activity panel appears as soon as the first `ToolStart` event lands
  (within ~200 ms of send); assistant bubble appears once `ToolResult` is
  consumed and the final text token streams. The order of `wait_for`s is:
  status-label тЖТ message-bubble.message--user тЖТ tool-activity тЖТ message-bubble.message--assistant.
- Token-metrics line ("X tokens ┬╖ Yms ┬╖ Z tok/s") is rendered inside the
  assistant bubble and shouldn't be asserted on numerically (varies per
  prompt + per run).
